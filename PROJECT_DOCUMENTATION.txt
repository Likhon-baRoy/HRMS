1. Project Overview
-------------------
This project is a web-based Human Resource Management System. The main goal is to help an HR department manage employees, salaries, payrolls, attendance, and user access.

The application has two main parts:

1. Backend API
   The backend is responsible for storing data, validating requests, applying business rules, generating payroll, and protecting routes with authentication and authorization.

2. Angular Frontend
   The frontend gives users a graphical interface where they can log in, view dashboards, manage employees, assign salaries, generate payroll, check attendance, and manage user roles.

The project is beginner-friendly but still follows useful software design practices such as separation of concerns, DTO usage, validation classes, services, reusable base classes, and role-based authorization.

3. Database and ER Design
-------------------------
The database is designed around employees. Most HRMS features connect back to the Employee table.

Main Entities:

Employee
- Stores employee personal and job information.
- Important fields: EmployeeCode, FirstName, LastName, Email, Phone, Address, DateOfBirth, HireDate, EmploymentType, EmployeeStatus, AccountNumber.
- Each employee belongs to one department.
- Each employee has one position.
- Each employee may have one user account.
- Each employee can have salary records, payroll records, and attendance records.

Department
- Stores department information.
- Important fields: Name, Description, ManagerId, RecordStatus.
- One department can have many employees.
- One department can have many positions.

Position
- Stores job position information inside a department.
- Important fields: Title, JobLevel, DepartmentId.
- One position can be assigned to many employees.

UserAccount
- Stores login and access permission information.
- Important fields: EmployeeId, Username, PasswordHash, Role, LastLoginAt.
- Role means access permission, for example Admin, HR, Manager, or Employee.
- Position is separate from role. Position means job title inside a department.

Salary
- Stores employee salary components.
- Important fields: EmployeeId, BasicSalary, HouseRent, MedicalAllowance, TransportAllowance, OtherAllowance, GrossSalary, EffectiveDate, EndDate, IsCurrent.
- One employee can have salary history, but only one salary is marked as current.

SalaryRevision
- Stores salary change history.
- Important fields: SalaryId, OldGrossSalary, NewGrossSalary, Reason, RevisionDate, ApprovedBy.
- Created when a salary is revised.

Payroll
- Stores generated monthly payroll data.
- Important fields: EmployeeId, PayrollMonth, PayrollYear, BasicSalary, GrossSalary, BonusAmount, DeductionAmount, TaxAmount, NetSalary, Status.
- Payroll is generated from current salary information.

Attendance
- Stores employee attendance records.
- Important fields: EmployeeId, AttendanceDate, CheckInTime, CheckOutTime, Status, Remarks.

Bonus, Deduction, PayrollDeduction
- These support payroll additions and deductions.


4. ER Relationship Summary
--------------------------
Employee -> Department
- Many employees belong to one department.
- Relationship: many-to-one.

Employee -> Position
- Many employees can have the same position.
- Relationship: many-to-one.

Department -> Position
- One department can have many positions.
- Relationship: one-to-many.

Employee -> UserAccount
- One employee can have one login account.
- Relationship: one-to-one.

Employee -> Salary
- One employee can have many salary records over time.
- Only one record should be current.
- Relationship: one-to-many.

Salary -> SalaryRevision
- One salary can have revision records.
- Relationship: one-to-many.

Employee -> Payroll
- One employee can have many monthly payroll records.
- Relationship: one-to-many.

Employee -> Attendance
- One employee can have many attendance records.
- Relationship: one-to-many.


5. Backend Architecture
-----------------------
The backend is organized in a layered style. This is not a very complex enterprise Clean Architecture setup, but it follows clean separation between responsibilities.

Important folders:

Controllers
- Receive HTTP requests from the frontend.
- Call service methods.
- Return API responses.
- Example: EmployeeController, SalaryController, PayrollController, AuthController.

Services
- Contain business logic.
- Example: EmployeeService manages employee creation, update, deletion, and filtering.
- Example: SalaryService manages current salary and salary revision rules.
- Example: PayrollService generates payroll based on current salaries.

DTOs
- Data Transfer Objects.
- Used to send and receive clean request/response data.
- Prevents directly exposing database models to the frontend.

Models
- Entity classes that represent database tables.
- Example: Employee, Department, Position, Salary, Payroll, UserAccount.

Data
- Contains AppDbContext, EF Core configurations, migrations, and database seeding.

Validators
- FluentValidation classes.
- Used to validate request DTOs before business logic runs.

Responses
- Contains common response structures like ApiResponse and PagedResult.

Middleware
- Contains ExceptionMiddleware.
- Converts exceptions into clean API error responses.


6. API Design Pattern
---------------------
The API follows REST-style endpoints.

Common methods:

GET
- Used to fetch data.
- Example: GET /api/employee

POST
- Used to create data or perform an action.
- Example: POST /api/employee
- Example: POST /api/payroll

PUT
- Used to update existing data.
- Example: PUT /api/employee/{id}

DELETE
- Used to remove or soft-delete data.
- Example: DELETE /api/employee/{id}

The controllers are kept simple. They do not contain large business logic. Instead, they call services.

Example flow:

Frontend form submits employee data.
Angular EmployeeService sends POST request to API.
EmployeeController receives request.
FluentValidation validates the DTO.
EmployeeService applies business rules.
Entity Framework saves data to PostgreSQL.
API returns a response to Angular.
Angular updates the UI.


7. Authentication and Authorization
-----------------------------------
Authentication means checking who the user is.

Authorization means checking what the user is allowed to do.

Authentication Process:

1. User enters username and password.
2. Angular sends login request to POST /api/auth/login.
3. Backend checks username and password hash.
4. If valid, backend generates a JWT token.
5. Angular stores the token in localStorage.
6. Every later API request sends the token in the Authorization header.
7. Backend reads the token and identifies the user.

Authorization:

The system uses roles:
- Admin
- HR
- Manager
- Employee

Examples:
- Admin can manage user accounts and roles.
- Admin and HR can manage salary and payroll.
- Manager can view dashboard, employees, and attendance.
- Employee can access profile and attendance.

Protected Admin Rule:

The seeded admin account is protected.
Its role cannot be changed.
Its user account cannot be deleted.
Its employee record cannot be terminated.

This prevents losing admin access by mistake.


8. Validation
-------------
Validation is handled with FluentValidation.

Examples:

Employee validation:
- Employee code is required.
- Email must be valid.
- Phone must have correct length.
- Department must exist.
- Position must exist.
- Employee must be at least 18 years old.

Salary validation:
- Employee must exist.
- Basic salary must be greater than zero.
- Allowances cannot be negative.
- Effective date is required.

Auth validation:
- Username is required.
- Password has minimum rules.
- Role must be valid.

Validation helps avoid bad data entering the database.


9. Error Handling
-----------------
The backend uses custom exceptions and middleware.

Important exceptions:
- NotFoundException
- AppValidationException
- AccountDisabledException

ExceptionMiddleware catches errors and returns a clean response.

Example error response:

{
  "success": false,
  "message": "Validation failed",
  "errors": {
    "email": ["Email already exists"]
  }
}

This makes frontend error display easier.

PaginationParams contains:
- Page

PagedResult returns:
- Items

Used in:

- Attendance list

Examples:

Employee list supports:

- Filter by department.
- Filter by employee status.

Salary list supports:
- Filter by employee.

Attendance list supports:
- Filter by employee.
- Filter by attendance date.


11. AutoMapper
--------------
AutoMapper is used to convert between models and DTOs.

Example:

Employee entity contains Department and Position objects.
EmployeeDto contains DepartmentName and PositionTitle.

AutoMapper helps convert this cleanly without writing repetitive manual mapping code everywhere.


12. Entity Framework Core
-------------------------
Entity Framework Core is used as the ORM.

Main responsibilities:
- Converts C# models into database tables.
- Handles relationships.
- Runs migrations.
- Performs queries.
- Saves changes.

AppDbContext contains DbSet properties for all main tables.

Examples:
- DbSet<Employee> Employees
- DbSet<UserAccount> UserAccounts
- DbSet<Salary> Salaries
- DbSet<Payroll> Payrolls

EF configurations define:
- Table names
- Required fields
- Maximum lengths
- Indexes
- Relationships
- Delete behavior

Global query filters are used for active records, so deleted or inactive records are hidden automatically in normal queries.

To assign and update employee salaries.

- Basic salary
- House rent
- Medical allowance
- Transport allowance
- Other allowance
- Gross salary
- Effective date

Salary revision:
When current salary is changed, a SalaryRevision record stores:
- Old gross salary
- New gross salary
- Reason
- Revision date
- Approved by user

-----------------------------

To generate monthly salary payments for employees.

Payroll uses current salary records.

Payroll calculates:
- Gross salary
- Bonus

- Tax
- Net salary

Formula:
Net Salary = Gross Salary + Bonus - Deduction - Tax

Features:
- Generate payroll by month and year.
- Avoid duplicate payroll for the same employee/month.
- Show payroll list.
- Show salary components and final net salary.

26. How the Project Works End-to-End
------------------------------------
Example: Creating an employee

1. Admin or HR opens Employee page.
2. User clicks Add Employee.
3. Angular opens employee form dialog.
4. User fills employee data.
5. Angular sends POST /api/employee.
6. Backend validates DTO.
7. EmployeeService saves employee.
8. API returns success response.
9. Angular refreshes employee table.

Example: Assigning salary

1. Admin or HR opens Salary page.
2. User clicks Assign Salary.
3. User selects employee and salary components.
4. Angular sends POST /api/salary.
5. Backend validates salary.
6. SalaryService creates current salary.
7. Angular refreshes salary list.

Example: Updating salary

1. Admin or HR edits existing salary.
2. Backend marks old salary as non-current.
3. Backend creates salary revision record.
4. Backend creates new current salary.
5. Salary history is preserved.

Example: Generating payroll

1. Admin or HR opens Payroll page.
2. User selects month/year and enters bonus, deduction, tax values if needed.
3. Angular sends POST /api/payroll.
4. PayrollService gets employees and current salaries.
5. System calculates net salary.
6. Payroll records are saved.
7. Payroll list displays generated records.

Example: Login and profile

1. User logs in.
2. API returns JWT token and role.
3. Angular stores token.
4. User opens My Profile.
5. Angular calls GET /api/auth/profile.
6. API returns current user's employee, role, position, department, and salary information.

29. Important Beginner Concepts Used
------------------------------------
DTO:
A DTO is a simple object used to transfer data between frontend and backend.

Service:
A service contains business logic. Controllers call services.

Controller:
A controller receives HTTP requests and returns responses.

Entity:
An entity represents a database table.

Migration:
A migration updates the database schema based on entity changes.

Validation:
Validation checks if input data is correct before saving.

JWT:
JWT is a token used to remember logged-in users.

Role:
A role controls what a user can access.

Pagination:
Pagination loads data page by page instead of loading everything at once.

Soft delete:
Soft delete means the record is not physically removed from the database. Its status is changed so it no longer appears in normal lists.
